# Anaplan Functions Reference for AI Financial Model Builders

**Source:** https://help.anaplan.com/all-functions-160769b0-de37-4f08-87a0-cc3aa55525a3

This document converts the Anaplan function catalogue into an AI-friendly modelling reference. It is intended to help an AI assistant choose the right function, understand the syntax, and apply functions in financial, supply-chain, sales, workforce, and operational planning models.

## How to use this reference

- Prefer simple, auditable formulas over dense nested formulas.
- Store mappings, rates, flags, and assumptions in SYS modules, then retrieve them with LOOKUP, SUM, SELECT only where appropriate.
- Use dimensionality deliberately. A function should move data between well-designed cubes; it should not compensate for poor dimensional design.
- Use DIVIDE for ratios to avoid divide-by-zero problems.
- Use time functions for forecast, prior-period, YTD/QTD/MTD, opening balance, closing balance, and cash timing logic.
- Use financial functions for DCF, loans, leases, bonds, and investment appraisal.
- Use text functions mostly for integration validation and data preparation; avoid text-heavy calculations in large cubes.

## AI formula selection rules

1. If moving data from one dimensionality to another, first identify source dimensions, target dimensions, and mapping line items. Then choose LOOKUP, SUM, or SELECT.
2. If calculating a percentage or ratio, use DIVIDE rather than `/` unless divide-by-zero is impossible and documented.
3. If referencing prior/future periods, prefer PREVIOUS/NEXT for adjacent periods, OFFSET for fixed shifts, and LAG/LEAD when the shift is a driver.
4. If building financial asset, debt, lease, or investment logic, consider PV, FV, PMT, IPMT, PPMT, RATE, NPER, NPV, and IRR before creating manual formulas.
5. If creating validations, use ISBLANK/ISNOTBLANK, ANY/ALL, and clear boolean line items instead of embedding validation inside numeric formulas.
6. If a formula needs hard-coded list items, challenge the design. Use system modules and mappings unless SELECT is genuinely structural and stable.
7. If a formula is long, split it into intermediate line items with clear names and summary methods.

## Notes for model builders

- Review summary methods for every line item. Incorrect summary settings are a common cause of wrong financial totals.
- Do not use subsidiary views as a substitute for proper module design.
- Avoid formula repetition. Create reusable SYS and CALC modules.
- Use codes for list imports and FINDITEM mappings.
- Keep calculations close to the lowest necessary dimensionality, then aggregate through hierarchies.
- Document assumptions and non-obvious function choices in line item notes.
